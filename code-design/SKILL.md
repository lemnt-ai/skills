# 需求开发设计文档生成

基于 `code_design_template.md` 模版生成标准化的需求开发设计文档。

## 快速决策树

| 需求类型 | 覆盖策略 | 章节完整性要求 |
|----------|----------|----------------|
| 模糊需求 | → Step 2 追问3个问题 | — |
| 新功能(涉及DB) | → 全15章，重点DDL+API+缓存+MQ | 全部章节完整输出 |
| 新功能(不涉及DB) | → 跳第7章，其他全量 | 跳章标注"无 DDL 变更" |
| 增量修改(加参/加字段) | → 仅输出变更章节，Step 4规划 | **但必须保留完整15章标题，未变更章节标注"本次无变更（原因：xxx）"** |
| 重构/优化(不改接口DB) | → 聚焦第2/4/5/12/14章 | **但必须保留完整15章标题，未聚焦章节标注"本次无变更（原因：xxx）"** |
| Bug修复 | → 聚焦第4+14章，最小化 | **但必须保留完整15章标题，未聚焦章节标注"本次无变更（原因：xxx）"** |

## 流程

### Step 1 — 模版解析
**输入:** `code_design_template.md`（本 skill 目录下）  
**输出:** 明确 15 个章节的结构和输出要求  

- 读取模板，理解全部章节的结构和输出规范
- ⚠️ 如果模版文件不存在：使用内置默认章节结构生成，文档开头标注"未使用模版文件"
- ⚠️ 如果模版版本与 skill 规范不一致：以 skill 章节输出规范为准，在文档中标注差异

### Step 2 — 需求理解
**输入:** 用户原始需求描述  
**输出:** 「需求理解摘要」（背景 + 影响范围 + 业务规则）  

- 梳理：需求背景、目标、类型、影响范围（接口/DB/服务）、业务流程和规则
- ⚠️ 如果用户需求描述模糊（如仅"做个新功能"），**必须主动追问 3 个关键问题**后再继续：
  1. 需求背景和目标是什么？
  2. 影响哪些模块/接口/数据库？
  3. 核心业务流程是什么？
- 🔴 **检查点**：输出「需求理解摘要」，请用户确认后再进入下一步。如有偏差，修正后继续。

### Step 3 — 源码分析
**输入:** 需求影响范围 + 项目源码路径  
**输出:** 「代码分析发现」（关键路径 + 冲突点 + 设计约束）  

- 深入分析相关模块的源码（Controller → Service → Domain → Mapper → API）
- ⚠️ 如果源码不可访问或分析超时：基于需求描述生成，在文档中标注"⚠️ 未经代码验证"并列出需人工确认的风险项
- ⚠️ 如果发现现有代码与需求冲突：在文档中标注冲突点，提供两种方案并说明 trade-off
- 🔴 **检查点**：展示「代码分析发现」，请用户确认后再继续。确认后进入设计输出阶段。

### Step 4 — 输出规划
**输入:** 需求理解摘要 + 代码分析发现  
**输出:** 「章节结构预览」（覆盖/跳过哪些章节及原因）  

- 根据需求类型（新增/优化/重构/修复），决定本次输出覆盖哪些章节、可以跳过哪些章节
- 增量修改只需覆盖变更涉及的章节
- 🔴 **检查点**：输出「章节结构预览」，请用户确认后再展开完整内容。确认后进入逐章输出阶段。

### Step 5 — 逐章生成
**输入:** 章节结构预览  
**输出:** 完整设计文档（按分组顺序输出）  

按分组顺序逐一输出（每组内按模板章节号顺序）：

| 分组 | 章节 | 重点 |
|------|------|------|
| **核心设计** | ①工作流 ②需求背景 ③需求范围 ④需求分析 ⑤架构设计 | 业务流程、状态机、风险点 |
| **接口数据** | ⑥API设计 ⑦数据库设计 ⑧枚举常量 | DDL、参数表、JSON 示例 |
| **集成设计** | ⑨缓存设计 ⑩MQ设计 ⑪安全设计 ⑫日志监控 | Key 格式、TTL、权限码 |
| **质量保障** | ⑬测试方案 ⑭验收标准 ⑮输出要求 | 用例≥3条、Given-When-Then |

- 每个章节输出完成后，对照章节输出规范（详见下方）自检
- 增量修改：仅输出变更涉及的章节，未变章节标注"本次无变更"

### Step 6 — 后置自检
**输入:** 完整设计文档草稿  
**输出:** 验证通过的设计文档  

执行生成后检查清单：

- [ ] 所有"本次不做"项是否标注了原因
- [ ] 风险点分析是否 ≥ 3 项
- [ ] 验收标准是否可量化/可测试
- [ ] **是否保留了完整15章骨架** — 未变更章节是否标注了"本次无变更（原因：xxx）"，而非直接跳过
- [ ] **代码引用是否包含精确文件路径和行号** — 格式 `Xxx.java:L123`，禁止仅写文件名
- [ ] 无变更章节是否明确表述，如"无 DDL 变更"
- [ ] API 参数表、请求示例、错误码是否完整
- [ ] DDL 语句是否包含索引和字段注释

### Step 7 — 生成实现计划
**输入:** 完整设计文档  
**输出:** 「实现计划」（文件清单 + 实现顺序 + 复杂度评估）  

- 将设计文档转化为可执行的实现计划，供 `code-generate` 按序使用
- 按数据层 → API层 → 业务层 → 接口层 → 配置的顺序列出
- 每个文件标注：精确路径、操作类型（新增/修改/删除）、复杂度（★/★★/★★★）
- 🔴 **检查点**：输出「实现计划预览」，确认文件清单和顺序无误后结束
- 计划模板详见下方「实现计划生成」章节

## 章节输出规范

每个章节按以下规范生成，确保输出可直接被 `code-generate` skill 作为输入使用。

### 1. 工作流（Workflow）

| 要求 | 说明 |
|------|------|
| 内容 | **开发实现流程** — 按数据层→API层→业务层→接口层→配置步骤，标注参与模块/服务 |
| 格式 | 泳道图或步骤序列，标注关键决策点 |
| 必含 | 涉及的模块/服务/角色 |
| 不推荐 | `[x] PENDING` 式 checklist（容易与设计任务进度混淆，应给出可执行的开发步骤） |

### 2. 需求背景（Requirement Background）

| 子章节 | 内容要求 |
|--------|----------|
| 业务背景 | 当前痛点、业务场景描述，引用具体业务数据或用户反馈 |
| 需求目标 | 量化目标（如"优化后RT < 200ms"），每项目标独立编号 |
| 需求类型 | 新增 / 优化 / 重构 / 修复 |

必含：用 AS-IS / TO-BE 对比说明。

### 3. 需求范围（Scope）

| 子章节 | 格式 |
|--------|------|
| 本次要做 | 列表形式，逐条描述，每条有唯一编号 |
| 本次不做 | 列表形式，每项标注不做的原因 |

### 4. 需求分析（Requirement Analysis）

| 子章节 | 输出规范 |
|--------|----------|
| 核心业务流程 | AS-IS / TO-BE 对比图，泳道图或时序图展示关键交互，标注参与方（用户/系统/外部） |
| 业务规则 | 状态机描述（状态流转表：当前状态→事件→下一状态）、校验规则、幂等策略 |
| 风险点分析 | ≥3项，每项格式：`[风险描述] → [影响] → [应对方案]` |

### 5. 整体架构设计（Architecture Design）

使用架构图或模块关系描述，标注新增/修改的模块。
格式：模块名 → 职责说明 → 依赖关系。

### 6. API 设计（API Design）

模板格式：
- `[POST/GET/PUT/DELETE] /api/v1/{resource}/{action}`
- 请求参数：| 参数名 | 类型 | 必填 | 说明 |
- 请求示例：JSON body
- 返回结果：`{ "code": 0, "message": "success", "data": {} }`
- 错误码：| 错误码 | HTTP状态码 | 说明 |

必含：每接口的参数表、请求示例、返回结果、错误码。

### 7. 数据库设计（Database Design）

```
表名：xxx（逻辑名）
DDL：
CREATE TABLE xxx (
  id bigint(20) NOT NULL AUTO_INCREMENT COMMENT '主键',
  ... -- 业务字段
  del_flag char(1) DEFAULT '0' COMMENT '删除标志',
  create_by/update_by/create_time/update_time -- BaseDO 字段
  PRIMARY KEY (id) USING BTREE,
  INDEX idx_xxx (yyy) USING BTREE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='表注释';
```

新增索引/变更字段：用表格列出。无变更 → 明确写"无 DDL 变更"。

### 8. 枚举 / 常量定义

```
枚举：XxxEnum
| 枚举值 | 编码 | 说明 |
|--------|------|------|
| XXX    | 0    | xxx  |

常量：XxxConstant
| 常量名 | 值 | 说明 |
|--------|-----|------|
| XXX_KEY | "xxx" | xxx |
```

### 9. 缓存设计（Cache Design）

| 缓存 | Key格式 | 数据类型 | TTL | 说明 |
|------|---------|---------|-----|------|
| xxx | `模块:业务:id` | String/Hash | 3600s | xxx |

必含：Key格式、TTL、缓存更新策略（主动更新/过期淘汰）。

### 10. MQ 设计（Message Queue）

| 消息 | Topic | 生产者 | 消费者 | 重试策略 | 说明 |
|------|-------|--------|--------|---------|------|
| XxxEvent | topic-xxx | 模块A | 模块B | 3次/死信 | xxx |

### 11. 安全设计（Security Design）

| 维度 | 输出要求 |
|------|----------|
| 权限码 | `@RequiresPermissions("模块:操作")`，逐接口列出 |
| 幂等 | Idempotency-Key + 接口名 + 幂等策略 |
| 防刷 | 用户级/IP 级限流阈值 + 时间窗口 |
| 数据权限 | 角色 → 数据范围规则 → 实现方式 |

### 12. 日志与监控（Observability）

| 维度 | 输出规范 |
|------|----------|
| 业务日志 | `log.info/error("xxx, key={}", val)` 占位符格式 |
| TraceId | 贯穿全链路，dsp-common-dubbo 自动透传 |
| 监控指标 | 指标名 `api.xxx.rt` + P99阈值 + 统计口径 |
| 告警规则 | 触发条件 + 通知方式（钉钉/短信/电话） |

### 13. 测试方案（Testing）

| 类型 | 覆盖范围 | 用例表头 |
|------|----------|----------|
| 单元测试 | 核心逻辑 + 异常分支 + 边界 | 测试项 \| 方法 \| 入参 \| 预期 |
| 集成测试 | 数据库/缓存/MQ | 测试项 \| 验证点 \| 前置条件 \| 预期结果 |
| E2E 测试 | 主业务流程 | 场景 \| 步骤 \| 验证点 |

必含：每种测试 ≥ 3 条用例，有明确入参和预期结果。仅覆盖变更范围。

### 14. 验收标准（Acceptance Criteria）

逐条列出，每条格式：
```
Given [前置条件]
When [操作]
Then [预期结果]
```

每条必须可量化、可测试。

### 15. 输出要求（Output Requirements）

- 文档路径：`doc/{module}-{feature}-design.md`
- 语言：中文（代码示例保持原语言）
- 引用代码：标注文件路径+行号，格式 `src/main/java/.../Xxx.java:L123`

# 实现计划生成

设计文档输出并确认后，自动生成**实现计划**，供 `code-generate` 按序执行。

## 计划结构

| # | 阶段 | 生成内容 | 依赖 |
|:-:|------|----------|:----:|
| 1 | **数据层** | DO + Mapper + XML (如有) | 无 |
| 2 | **API层** | BO + VO + Converter + Dubbo接口 | 数据层 |
| 3 | **业务层** | Service接口 + ServiceImpl | API层 |
| 4 | **接口层** | Controller | 业务层 |
| 5 | **配置** | 权限SQL + Nacos配置 + MQ Topic | 接口层 |

## 输出格式

```markdown
## 实现计划

### Phase 1: 数据层
- [ ] 创建 `domain/Xxx.java` — 继承 BaseDO，Δ 行数
- [ ] 创建 `mapper/XxxMapper.java` — 继承 BaseMapper

### Phase 2: API层
- [ ] 创建 `api/.../bo/XxxBo.java` — 继承 BaseBO
- [ ] 创建 `api/.../vo/XxxVo.java` — 继承 BaseVO
- [ ] 创建 `converter/XxxConverter.java` — MapStruct

### Phase 3: 业务层
- [ ] 创建 `service/XxxService.java` — 接口
- [ ] 创建 `service/impl/XxxServiceImpl.java` — 实现

### Phase 4: 接口层
- [ ] 创建 `controller/XxxController.java`
    - POST /api/v1/xxx — @RequiresPermissions("xxx:add")
    - GET /api/v1/xxx/{id} — @RequiresPermissions("xxx:query")

### 变更清单
| 文件 | 操作 | 估算复杂度 |
|------|------|:----------:|
| Xxx.java | 新增 | ★☆☆ |
| XxxServiceImpl.java | 新增 | ★★☆ |
| application.yml | 修改 | ★☆☆ |
```

必含：每个文件的精确路径、操作类型（新增/修改/删除）、复杂度评估（★/★★/★★★）。
